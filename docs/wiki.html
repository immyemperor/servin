Configuration Overview

🎯 Configuration Files

Servin uses YAML configuration files to manage runtime settings, daemon options, registry configurations, and network settings. Configuration can be specified via files, environment variables, or command-line flags.

Configuration File Locations

🪟 Windows
  • System: C:\ProgramData\Servin\config.yaml
  • User: %USERPROFILE%\.servin\config.yaml
  • Working Dir: .\servin.yaml
Highest to lowest priority
🐧 Linux
  • System: /etc/servin/config.yaml
  • User: ~/.config/servin/config.yaml
  • Working Dir: ./servin.yaml
Highest to lowest priority
🍎 macOS
  • System: /etc/servin/config.yaml
  • User: ~/Library/Application Support/servin/config.yaml
  • Working Dir: ./servin.yaml
Highest to lowest priority

Main Configuration File

📄 config.yaml
# Servin Configuration File
# /etc/servin/config.yaml

# Daemon settings
daemon:
  # Data directory for containers, images, volumes
  data_root: "/var/lib/servin"
  
  # Runtime configuration
  runtime: "runc"
  runtime_args: []
  
  # Logging configuration
  log_level: "info"
  log_file: "/var/log/servin/servin.log"
  log_format: "text"  # text or json
  
  # Process limits
  max_concurrent_downloads: 6
  max_concurrent_uploads: 6

# Network configuration
network:
  # Default network for containers
  default_network: "bridge"
  
  # Bridge network settings
  bridge:
    name: "servin0"
    subnet: "172.17.0.0/16"
    gateway: "172.17.0.1"
    ip_masq: true
    icc: true
    mtu: 1500

# Storage configuration
storage:
  # Storage driver
  driver: "overlay2"
  
  # Storage options
  opts:
    overlay2.override_kernel_check: "true"
    overlay2.size: "120G"
  
  # Image storage settings
  image_store: "/var/lib/servin/images"
  
  # Volume storage settings
  volume_store: "/var/lib/servin/volumes"

# Registry configuration
registries:
  # Default registry
  default: "docker.io"
  
  # Registry mirrors
  mirrors:
    "docker.io":
      - "https://mirror.example.com"
  
  # Insecure registries (HTTP)
  insecure:
    - "registry.internal:5000"
  
  # Registry authentication
  auths:
    "registry.example.com":
      username: "user"
      password: "pass"

# CRI configuration
cri:
  # Enable CRI server
  enabled: true
  
  # CRI socket path
  socket_path: "/var/run/servin/servin.sock"
  
  # CRI server settings
  server:
    address: "127.0.0.1"
    port: 10010
    
  # Image service configuration
  image_service:
    max_parallel_downloads: 6
    
  # Runtime service configuration
  runtime_service:
    cgroup_manager: "systemd"
    default_runtime: "runc"

# API server configuration
api:
  # Enable REST API server
  enabled: true
  
  # API server settings
  server:
    address: "0.0.0.0"
    port: 8080
    
  # TLS configuration
  tls:
    enabled: false
    cert_file: "/etc/servin/tls/server.crt"
    key_file: "/etc/servin/tls/server.key"
    
  # Authentication
  auth:
    method: "none"  # none, basic, token, jwt
    token_file: "/etc/servin/tokens.json"

# Metrics and monitoring
metrics:
  # Enable metrics endpoint
  enabled: true
  
  # Metrics server settings
  server:
    address: "127.0.0.1"
    port: 9090
    path: "/metrics"

🔧 Troubleshooting

Common Issues

⚠️ Installation Problems

NSIS not found

Error: NSIS not installed!

Solution:

  1. Download NSIS from official site
  2. Install latest version
  3. Verify: makensis.exe in PATH

CGO compilation failed

cgo: C compiler "gcc" not found

Solution (Windows):

  1. Install MinGW-w64 UCRT64 toolchain
  2. Add C:\mingw64\bin to PATH
  3. Verify: gcc --version
🚫 Runtime Issues

Permission denied

Error: permission denied: cannot connect

Solution:

  • Linux/macOS: Run with sudo or add user to servin group
  • Windows: Run as Administrator

Port already in use

Error: bind: address already in use (port 10010)

Solution:

# Find process using port
netstat -tlnp | grep 10010    # Linux
netstat -ano | findstr 10010  # Windows

# Use different port
servin daemon --cri-port 10011

🔌 Container Runtime Interface (CRI)

Kubernetes Integration

🎯 CRI Compatibility

Servin implements the Kubernetes Container Runtime Interface (CRI) v1alpha2, enabling seamless integration with Kubernetes clusters. Use Servin as a drop-in replacement for containerd or CRI-O for enhanced container management.

CRI Server Configuration

🚀 Starting CRI Server
# Start CRI server with default settings
servin daemon --cri

# Custom CRI endpoint
servin daemon --cri --cri-endpoint unix:///var/run/servin/servin.sock

# Custom port for HTTP CRI
servin daemon --cri --cri-port 10010

# CRI with additional logging
servin daemon --cri --log-level debug

# Start as system service
sudo systemctl start servin-cri
⚙️ Configuration Options
# Configuration file: /etc/servin/config.yaml
cri:
  enabled: true
  endpoint: "unix:///var/run/servin/servin.sock"
  port: 10010
  image_service:
    registry_config_path: "/etc/containers/registries.conf"
  runtime_service:
    default_runtime: "runc"
    runtimes:
      runc:
        path: "/usr/bin/runc"

func TestContainerLifecycle(t *testing.T) {
    // Setup test environment
    testEnv := NewTestEnvironment(t)
    defer testEnv.Cleanup()
    
    client := testEnv.Client()
    
    // Test container creation
    container, err := client.ContainerCreate(context.Background(), &ContainerConfig{
        Image: "alpine:latest",
        Cmd:   []string{"echo", "hello"},
    })
    require.NoError(t, err)
    
    // Test container start
    err = client.ContainerStart(context.Background(), container.ID)
    require.NoError(t, err)
    
    // Wait for completion
    statusCh, errCh := client.ContainerWait(context.Background(), container.ID)
    select {
    case status := <-statusCh:
        assert.Equal(t, int64(0), status.StatusCode)
    case err := <-errCh:
        t.Fatalf("Container wait failed: %v", err)
    }
    
    // Cleanup
    err = client.ContainerRemove(context.Background(), container.ID)
    require.NoError(t, err)
}

❓ Frequently Asked Questions

Q: Is Servin compatible with Docker?

Yes! Servin provides Docker-compatible APIs and command-line interface. You can use most Docker commands directly with Servin, making migration seamless.

Q: Can I use Servin with Kubernetes?

Absolutely! Servin implements the Container Runtime Interface (CRI) v1alpha2, making it fully compatible with Kubernetes clusters as a container runtime.

Q: What platforms does Servin support?

Servin supports Windows 10+, Linux (Ubuntu, CentOS, Debian, Fedora, Arch), and macOS 10.12+. We provide native installers for all platforms.

Q: How do I migrate from Docker to Servin?

Migration is straightforward:

  1. Install Servin using the installer wizard
  2. Export your Docker images: docker save
  3. Import to Servin: servin images load
  4. Update your scripts to use servin instead of docker

Q: Does Servin support Docker Compose?

Docker Compose support is planned for a future release. Currently, you can use Servin with individual container commands or Kubernetes manifests.

Q: How do I backup my containers and images?

Use these commands:

# Backup images
servin images save -o backup.tar image1:tag image2:tag

# Export container
servin containers export container_id > container_backup.tar

# Save volumes (copy data directories)
cp -r /var/lib/servin/volumes/myvolume/_data /backup/
